# Password recovery via Supabase recovery session, no router added

The login form's "forgot password" stub ([ADR-0004](0004-supabase-auth-scaffolding.md)) is now wired to Supabase's built-in reset flow. Clicking it swaps the login form for a `ForgotPasswordForm` (in `AuthPanel`) that calls `resetPasswordForEmail`, showing a "check your email" confirmation on success instead of navigating anywhere.

Supabase's reset email links back to the app with recovery tokens in the URL fragment; `supabase-js` picks these up on load (`detectSessionInUrl`, on by default) and establishes a session, firing a `PASSWORD_RECOVERY` event on `onAuthStateChange` rather than the usual `SIGNED_IN`. Without distinguishing this from an ordinary login, the user would land on whatever screen they last had open, generally logged in with no way to set a new password.

The app has no router — `App.tsx` picks screens via a plain `useState<Screen>`, so there's no path for a deep link to match. Rather than introduce routing for this one case, `useAuth` exposes an `isPasswordRecovery` flag (set on `PASSWORD_RECOVERY`, cleared on `SIGNED_OUT`) and `App` checks it before rendering anything else, short-circuiting straight to `ResetPasswordScreen` regardless of the current screen state. Submitting a new password there signs the user out (clearing `isPasswordRecovery` and dropping back to the normal app, unauthenticated) so they log in again with the new password through the ordinary `LoginForm` — satisfying "logging in with the new password succeeds" as a real, separate step rather than assuming the recovery session is good enough to leave them logged in.
