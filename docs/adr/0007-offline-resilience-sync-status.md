# Offline resilience: persisted retry queue + live sync status

Both preset and history pushes ([ADR-0005](0005-preset-sync.md), [ADR-0006](0006-history-sync.md)) previously swallowed failures outright — a push made while offline, or one that hit a transient Supabase error, was dropped for good and only reconciled on the next login-triggered pull. `pushPreset`/`pushTombstone`/`pushHistoryRecord` now route through a small generic retry queue (`retryQueue.ts`, one instance per resource: `presetPushQueue`, `historyPushQueue`) instead of a bare `try/catch`.

A queue persists its pending items to localStorage (so a page reload doesn't lose them), attempts every queued item whenever it's told to flush, and on failure keeps the item queued and schedules a retry with exponential backoff (capped) rather than dropping it. A flush is triggered on every `push()` call, on the browser's `online` event, and by its own backoff timer — there's no polling. Pushing an item with an id already queued replaces the older entry rather than duplicating it, so a preset edited twice while offline only ships its latest version. While offline (`navigator.onLine === false`), the queue doesn't even attempt a push — it waits for the `online` event — so failed-but-unattempted items read as "pending" rather than "retrying". Because a flush attempts every queued item together, the next backoff delay is based on whichever item has failed the *fewest* times, not the most — a freshly-failed item queued alongside one that's failed repeatedly shouldn't inherit the older item's much longer wait before its own next try.

This also fixes a latent bug: the old push functions only treated a *thrown* Supabase-client error as a failure, not a resolved `{ error }` response (the actual shape most Supabase failures — RLS violations, constraint errors — take). Both are now treated as failures and queued for retry.

`useSyncStatus` combines both queues' statuses (`idle` / `pending` / `retrying`) into a single `'synced' | 'pending' | 'failed-retrying'` value via `useSyncExternalStore`, gated behind `useAuth`'s status (`null` — render nothing — when logged out, since sync only runs while authenticated). `AuthPanel` renders this live via `SyncStatusIndicator`. Because queued items carry the `userId` active when they were enqueued, `useAuth` clears both queues whenever the session goes away (explicit sign-out, expiry, or a fresh unauthenticated load) so a stale push can't later land — or fail an RLS check — under a different account signed into the same device.

This further qualifies [ADR-0005](0005-preset-sync.md) and [ADR-0006](0006-history-sync.md): a push is no longer best-effort-and-forgotten while logged in — it's best-effort-and-retried, with local storage still the source of truth throughout. Logged-out behavior, and behavior when Supabase isn't configured, are unchanged.
